## Domain and Stack

A Python data pipeline that produces 1X2 / Over-Under / advancement probabilities and
**expected-value betting recommendations** for FIFA World Cup 2026 matches, plus a static
dashboard. Language: Python 3 (stdlib + `numpy`, `requests`). No framework; a thin Flask
`app.py` serves a template, and `src/pipeline/export_static.py` writes a self-contained
`docs/index.html`. Entry point: `python -m src.pipeline.daily_matchday_run`.

Data sources (all optional, graceful degradation): Polymarket (Gamma + Data API, public),
The Odds API (bookmakers), Kalshi, ESPN, FBref, eloratings.net, StatsBomb (API or local
open-data clone), a venue/weather client. Primary output: per-run JSON snapshot
(`data/snapshots/<ts>.json`), `data/processed/dashboard_data.json`, Obsidian markdown in
`memory/`, and `docs/index.html`.

Actors are all systems/automation: the daily batch job, external HTTP APIs, and a human
reading the dashboard / value-bet list. The highest-risk surface is the **probability →
uncertainty → value-bet** chain, because errors there are silent (a plausible-looking
recommendation that is quantitatively wrong) and carry real money meaning.

## Architecture

- `src/config.py` — paths, model weights, Elo snapshot, team aliases/codes, API-key loading, model-version fingerprint.
- `src/data_sources/*` — one client per external source. Each returns a status-tagged dict and degrades to `unavailable`/static fallback rather than raising.
- `src/model/` — the math: `features.py` (λ engines + Poisson/Dixon-Coles), `monte_carlo.py` (weighted MC), `ensemble.py` (source blending), `value_betting.py` (EV/Kelly/portfolio), `data_quality.py` (traffic-light gate), `calibration.py` (Brier/LogLoss/RPS/ROI/CLV), `knockout.py`, `whale_scoring.py`, plus `injuries`, `weight_optimizer`, `closing_loop`, `backtest`.
- `src/pipeline/daily_matchday_run.py` — orchestrator: discover → snapshot → features → ensemble → MC → value → quality gate → calibration → export.

Data flow: Polymarket discovery → per-match snapshot → `features.attack_defense_lambdas`
(Elo+StatsBomb+form+venue) and market/books/kalshi/whale probabilities → `ensemble.blend_probs`
+ `blend_lambdas` → `monte_carlo.simulate` → `value_betting.evaluate_match` →
`data_quality.score_match` → `apply_quality_gate` → portfolio aggregation → persist + export.

Most complex module: `features.py` (three λ engines, Dixon-Coles, grid-search inversion).
Most fragile: `value_betting.py` + `monte_carlo.py` (the uncertainty plumbing feeds the
money-facing value flags).

## Open Exploration Findings

- **OF-1 — MC "uncertainty bands" measure simulation noise, not parameter uncertainty.**
  `monte_carlo.py:3-6` (docstring) promises "5%/95%-Quantile … über Parameter-Resamples".
  But `simulate` draws an **independent** Gamma λ for *every* run (`monte_carlo.py:30-31`),
  then partitions runs into 200 contiguous blocks and takes the within-block 1X2 frequency
  (`monte_carlo.py:96-105`). Because each run already has its own λ, a block is a *mixed-λ*
  sample of size `chunk≈100`, so the across-block dispersion is dominated by MC sampling
  noise and **shrinks like 1/√chunk as `MC_RUNS` grows**. True parameter uncertainty
  (CV=0.12) would not shrink with more runs. This band feeds `value_betting` (see OF-4),
  so a `MC_RUNS` change silently moves betting recommendations. (Trace: `simulate` →
  `bands` → `evaluate_match` `band` → `_row` `band_width` → `estimate_sigma`.)

- **OF-9 — `monte_carlo.simulate` raises `ValueError` for `runs < 200`.**
  `monte_carlo.py:97-99`: `nb=200`, `chunk=max(1, runs//nb)`. For `runs=100`, `chunk=1`,
  `nb*chunk=200`, but `np.arange(runs)[:200]` yields only 100 elements →
  `.reshape(200,1)` raises. No test uses `runs<200` (default 20000), so this is latent. In
  the pipeline it would be swallowed by the per-match `try/except`
  (`daily_matchday_run.py:186-189`) → the match is **silently dropped**.

- **OF-3 — Integer Over/Under line: push mass mis-assigned to "Under".**
  `value_betting.py:143-145`: `floor=int(ln)`, `p_over = 1 - sum(total_dist[:floor+1])`,
  `p_under = 1 - p_over`. For a whole-number line (e.g. 3.0) exact `total==3` is a **push**
  (stake returned), but the code folds `P(total==3)` entirely into "Under". Harmless for the
  usual `.5` lines; wrong if any integer total is offered.

- **OF-4 — Totals value-bets use a default low σ, making them easier to flag than 1X2.**
  `evaluate_match` passes `band_width=None` for totals (`value_betting.py:148-150`), so
  `estimate_sigma` falls back to `bw=0.05` (`value_betting.py:38`). 1X2 rows pass a real MC
  band (typically wider). Lower σ ⇒ higher `conservative_prob` ⇒ `is_value` triggers more
  readily for Over/Under than for 1X2 at equal edge. (Trace: `evaluate_match` → `_row` →
  `estimate_sigma` → `conservative_prob`.)

- **OF-2 — Whale exposure mixes share-count and USD in one sum.**
  `whale_scoring.py:97-104` adds holder `amount` (Yes **share count**) while
  `whale_scoring.py:106-113` adds trade `size*price` (**USD**) into the *same*
  `exposure[w][outcome]`. The combined value is then both the ranking key
  (`whale_scoring.py:116`) and the probability weight (`weighted[k] += score*v`,
  `whale_scoring.py:124-125`). Two heterogeneous units summed as if comparable.

- **OF-5 — Team matching by substring can produce false matches.**
  `calibration.py:31-33` and `odds_client.py:117-120` both use
  `ca == cb or ca in cb or cb in ca` after canonicalization. Used to align forecasts to
  results (`calibration.py:146-148`) and to find odds events (`odds_client.py:123-128`). A
  team whose canonical name is a substring of another (or a partial alias slipping through)
  would bind to the wrong fixture and silently corrupt Brier/ROI or pull the wrong odds.

- **OF-6 — Market-agreement quality factor ignores draw/away disagreement.**
  `data_quality.py:24-35` computes dispersion over **only** `probs["team1_win"]` across
  sources. Two sources can agree on home-win yet disagree sharply on draw vs away and still
  score "high agreement", inflating the data-quality tier (which scales stakes).

- **OF-7 — `blend_lambdas` discards book/kalshi goal structure when Polymarket is absent.**
  `ensemble.py:74-89`: `cands` is built only from `market` and `model`; `books`/`kalshi`/
  `whale` weight is *folded into the market λ weight* (`ensemble.py:87-88`). But
  `blend_probs` *does* include `books`/`kalshi` in the probability blend
  (`ensemble.py:20-23,43`). So when `market` (Polymarket) is missing but books are present,
  the **probabilities use books but the MC λ uses only the model** — the book-implied goal
  structure is dropped though it drove the headline probabilities. (Cross-function:
  `blend_probs` vs `blend_lambdas`.)

- **OF-8 — `ensemble.blend_probs` whale-weight default disagrees with config.**
  `ensemble.py:33` uses `weights.get("whale", 0.2)`, but `config.ENSEMBLE_WEIGHTS["whale"]`
  is `0.15` (`config.py:45`). A caller passing custom weights without a `whale` key silently
  gets `0.2`, inconsistent with the documented default.

## Quality Risks

- **QR-1 (from OF-4) — Systematic over-recommendation of Over/Under bets.** A betting model
  that understates uncertainty on one market subset will over-flag that subset as "value".
  Open `value_betting._row` for a totals row vs a 1X2 row with the same edge and confirm the
  totals row reaches `is_value` at a lower true confidence. Failure mode: stake is allocated
  to Over/Under preferentially for a *plumbing* reason, not an edge reason.

- **QR-2 (from OF-1) — Betting output depends on `MC_RUNS`.** Because the band feeds σ which
  feeds `is_value`, raising `MC_RUNS` narrows bands → shrinks σ → more value flags. A
  performance/config knob silently changes money-facing output. Check: run `simulate` at
  `runs=20000` vs `200000` for the same λ and compare `uncertainty` band widths; they should
  be stable if they encode parameter uncertainty, but will shrink.

- **QR-3 (from OF-9) — Silent match loss on small-`runs` calls.** Any future call to
  `simulate(..., runs<200)` (debugging, a faster "draft" mode, a backtest sweep) raises and,
  inside the pipeline, the whole match is dropped via the broad `except`
  (`daily_matchday_run.py:186-189`) with only a console line. Check: `simulate(1.3,1.3,runs=50)`.

- **QR-4 (from OF-2) — Distorted whale signal / mislabeled specialists.** The unit mix skews
  which wallets rank top and the per-outcome weights. Blast radius is bounded by the
  `WHALE_SHIFT_CAP=0.08` gate in `ensemble.py:50-63`, but the displayed whale probabilities,
  ranking, and `football_specialist` flag (`whale_scoring.py:81`) are still wrong. Check:
  a holder with large share `amount` but tiny USD trades vs a trader with large USD.

- **QR-5 (from OF-5) — Silent corruption of the model's own scorecard.** A substring
  mis-match in `calibration._team_eq` binds a forecast to the wrong result, so Brier/LogLoss/
  RPS/ROI are computed against the wrong outcome — the very metrics used by
  `weight_optimizer.suggest_weights`. Check the 48-team field for canonical-name substring
  collisions and partial aliases reaching `_team_eq`.

- **QR-6 (from OF-3) — EV mis-estimate on integer totals.** If The Odds API returns a whole
  number total, `p_under` absorbs the push probability, biasing the Under EV upward and the
  Over EV. Check `evaluate_match` with a totals line of `3.0`.

  (`features.py:43`), MC weights `np.clip(w,1e-9,None)` (`monte_carlo.py:46`), Kelly/stake
  caps (`value_betting.py:25-32,63`), portfolio caps (`value_betting.py:217-235`).
- Outcome key set `("team1_win","draw","team2_win")` is the canonical enum, consistently
  used across `features`, `ensemble`, `monte_carlo`, `calibration`, `value_betting`. Verified
  the whale `holders`/`trades` snapshot is keyed by the same enum
  (`polymarket_client.snapshot_market:178-184`), so `whale_signal`'s `if k in weighted`
  guard (`whale_scoring.py:124`) does **not** silently drop — earlier hypothesis dismissed.
- Quality-gate state machine: `score → tier ∈ {green,amber,red} → stake_multiplier
  {1.0,0.5,0.0}` (`data_quality.py:62-66`); red suppresses value flags
  (`value_betting.py:206-213`).

## Pattern Deep Dive — Fallback and Degradation Path Parity

The probability path and the goal-expectation (λ) path must degrade *together*, but they
don't. `ensemble.blend_probs` (`ensemble.py:13-67`) admits five sources
(market/books/kalshi/model/whale), renormalizes active weights, and produces blended 1X2.
`ensemble.blend_lambdas` (`ensemble.py:70-102`) is supposed to produce the matching goal
expectations for the MC, but it only constructs candidates from `market` and `model`
(`ensemble.py:74-81`) and folds books+kalshi+whale weight into `w_market`
(`ensemble.py:87-88`). The hidden assumption is "Polymarket market is always present".
When it is **not** (Polymarket discovery failed, or a match exists only on the books feed),
`cands["market"]` is absent, so `parts` contains only the model; the book-implied λ — which
*did* shape `blend_probs` — never reaches the MC. Result: headline probabilities and the MC
scoreline/totals distribution are computed from inconsistent inputs. This is a genuine
prob-path/λ-path parity break (OF-7 / CB-3). Compare the symmetric, correct degradation in
`data_quality._market_agreement`, which at least *enumerates* market/books/kalshi
(`data_quality.py:27`) — though it then only reads `team1_win` (OF-6).

## Pattern Deep Dive — Cross-Implementation Consistency

`odds_client` parses the same h2h payload twice. `best_prices_for_match`
(`odds_client.py:153-182`) classifies each outcome via `_team_match(home,nm)` then flips
with `flipped`, builds `sharp` from Pinnacle-or-median, and devigs with **Shin**
(`odds_client.py:182`). `probs_for_match` (`odds_client.py:246-272`) re-implements the same
classification with `home`/`away` keys and flips at the end (`odds_client.py:277-278`), also
Shin. Two parsers of one contract = drift risk; they already differ in that totals consensus
uses **plain** `_devig` (`odds_client.py:216`) while h2h uses Shin — defensible, but
undocumented asymmetry. The two λ engines (`features.model_lambdas` vs
`attack_defense_lambdas`) are a second instance: only `attack_defense_lambdas` is wired into
the pipeline (`daily_matchday_run.py:136`); `model_lambdas` is effectively dead and uses a
different split (`(w/(1-w))**0.7` vs an Elo-per-goal GD model), so the two can silently
diverge in behavior/tests.

## Pattern Deep Dive — Spec-Structured Parsing Fidelity

`extract_1x2` reads `json.loads(outcomePrices)[0]` as the **Yes** price
(`polymarket_client.py:101`), assuming Polymarket always orders `[Yes, No]`. If a market ever
lists `[No, Yes]`, every probability inverts with no error. Team→market assignment uses
substring tests `raw1.lower() in q` (`polymarket_client.py:108-111`) after checking `"draw"`
first — robust for the current field but fragile to a team name embedded in the opponent's
or the question's text. `parse_slug` (`polymarket_client.py:55-67`) hard-codes `parts[2:5]`
as the date and requires `len(parts) >= 5`; a slug with a hyphenated code or extra segment
would mis-slice. `odds_client` totals parsing keys lines by `str(point)`
(`odds_client.py:205`), so `2.5` and `2.50` would be distinct buckets if the API ever varies
formatting. None of these are confirmed live failures, but each is an unguarded structural
assumption.

## Derived Requirements

- **REQ-001** (Tier 3) — `monte_carlo.simulate` must return normalized 1X2 (`sum≈1`) and consistent goal/score distributions for any `runs ≥ 1`, without raising. Basis: `monte_carlo.py:107-122`; conditions of satisfaction: no exception for `runs ∈ {1,50,199,20000}`; `score_matrix` sums to 1. (CB-1)
- **REQ-002** (Tier 4) — The `uncertainty` band must represent **parameter** uncertainty (CV=`MC_PARAM_UNCERTAINTY`) and be approximately invariant to `MC_RUNS`. Basis: docstring `monte_carlo.py:3-6`. (CB-2)
- **REQ-003** (Tier 3) — When the Polymarket `market` is absent but `books`/`kalshi` are present, the MC λ must reflect the same sources that drove `blend_probs`. Basis: `ensemble.py:13-102`. (CB-3)
- **REQ-004** (Tier 3) — Over/Under probabilities must respect push semantics on integer lines; `p_over + p_under + p_push = 1`. Basis: `value_betting.py:132-153`. (CB-4)
- **REQ-005** (Tier 3) — σ used for value detection must be derived comparably across markets; a market without an MC band must not receive a systematically smaller σ than one with a band at equal edge. Basis: `value_betting.py:35-39,66-91`. (CB-5)
- **REQ-006** (Tier 3) — Whale exposure aggregation must combine commensurable units (USD), not share-count plus USD. Basis: `whale_scoring.py:85-130`. (CB-6)
- **REQ-007** (Tier 3) — Team identity resolution must avoid false positives from substring containment for the 48-team field. Basis: `calibration.py:30-33`, `odds_client.py:117-120`. (OF-5)
- **REQ-008** (Tier 3) — `data_quality` market-agreement must consider all three outcomes, not only `team1_win`. Basis: `data_quality.py:24-35`. (OF-6)
- **REQ-009** (Tier 3) — Ensemble weight defaults referenced in code must equal `config.ENSEMBLE_WEIGHTS`. Basis: `ensemble.py:33` vs `config.py:45`. (OF-8)
- **REQ-010** (Tier 3) — Value flags must be suppressed under a red data-quality tier and stakes scaled by `stake_multiplier`. Basis: `value_betting.py:198-214`, `data_quality.py:62-66`. (regression guard; currently satisfied)
- **REQ-011** (Tier 3) — `extract_1x2` must correctly identify the Yes price irrespective of Polymarket outcome ordering. Basis: `polymarket_client.py:98-119`. (parsing fidelity)
- **REQ-012** (Tier 3) — Calibration must only use pre-kickoff snapshots (no leakage) and be timezone-correct. Basis: `calibration.py:152-158`; already covered by `test_calibration_timezone_and_leakage`.

## Derived Use Cases

- **UC-01** — Daily operator runs the pipeline and gets per-match 1X2/scoreline/totals probabilities with an uncertainty band. (REQ-001, REQ-002)
