# Spec Audit Protocol — Council of Three

> Generated by [Quality Playbook](https://github.com/andrewstellman/quality-playbook) v1.5.6 — Andrew Stellman
> Date: 2026-06-14 · Project: wm-dashboard-fussball-wahrscheinlichkeit

Three independent auditors review the code against `quality/REQUIREMENTS.md`. Each performs a
full audit; assign secondary emphasis lenses to bias attention without restricting coverage.

## Pre-audit docs validation
No `reference_docs/`. Auditors use Tier-3 (code) and Tier-4 (module docstrings + `memory/*.md`)
as the baseline. The docstrings in `monte_carlo.py`, `ensemble.py`, `value_betting.py` state
explicit contracts and are the primary reference for divergence findings.

## Audit prompt (copy-paste per auditor)
"Review this World Cup probability/value-betting pipeline against the requirements. Report
divergences between documented intent (docstrings/requirements) and code behavior with exact
file:line. A code-path trace is sufficient evidence. Do not flag self-documenting placeholders
or intentional, annotated design choices. Severity: credential/auth/injection always high; a
silent quantitative error in the probability→value chain is high if it changes a recommendation."

## Secondary emphasis lenses
- **Auditor 1 — numerical semantics:** MC band meaning (REQ-002), σ comparability (REQ-005),
  Dixon-Coles correctness.
- **Auditor 2 — data fusion & parity:** prob↔λ parity (REQ-003), whale units (REQ-006), agreement
  factor (REQ-008), weight defaults (REQ-009).
- **Auditor 3 — robustness & identity:** crash boundary (REQ-001), team matching (REQ-007),
  parsing fidelity (REQ-011), push semantics (REQ-004).

## Scrutiny areas (project-specific)
1. Does any documented contract (docstring) diverge from the implementation? (Expect REQ-002, REQ-003.)
2. Does any config-coupled output silently depend on a performance knob? (REQ-002 / `MC_RUNS`.)
3. Are heterogeneous quantities summed without unit reconciliation? (REQ-006.)
4. Can identity resolution bind the wrong fixture? (REQ-007.)

## Triage
Merge by confidence. Any minority finding (1/3) gets a re-investigation with a verification probe
that produces an executable assertion + line-number citation (write to
`quality/spec_audits/triage_probes.sh`). Confirmed code bugs are appended to the cumulative BUG
tracker in `quality/PROGRESS.md` with source "Spec Audit", then synced into `quality/BUGS.md`.
