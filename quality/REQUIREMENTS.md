## Overview

This project is a daily batch pipeline that turns public prediction-market prices, bookmaker
odds, and football statistics into **calibrated match probabilities and conservative
expected-value betting recommendations** for the 2026 FIFA World Cup, plus a phone-friendly
static dashboard. Every consumer is automation or a single human operator reading the output;
there is no multi-user surface. What the operator depends on is *quantitative trust*: a number
labelled "value" must reflect a real edge, an uncertainty band must mean what it says, and the
calibration scorecard must be computed against the right results. The highest-risk area is the
probability → uncertainty → value-bet chain, because every defect there is silent — the output
still looks plausible.

Requirements are Tier 3 (the code is the spec) or Tier 4 (the unusually precise module
docstrings and `memory/*.md`). No `reference_docs/` were supplied, so there are zero Tier 1/2
citations; this is a clean "spec-gap" degradation, not a failure.

## Functional section: Monte-Carlo & uncertainty

The MC engine must produce internally consistent distributions and an uncertainty band whose
*meaning* matches its documented contract and its downstream use in value detection.

### REQ-001 — Simulation robustness and normalization (Tier 3, specific)
- **Requirement:** `monte_carlo.simulate` must, for any `runs ≥ 1`, return without raising and
  yield normalized 1X2 (`sum ≈ 1`) plus goal/score distributions that each sum to ≈ 1.
- **User story:** As the daily job (or a backtest sweep) calling `simulate` with any run count,
  I expect a result, not an exception, *so that* a small/draft run never silently drops a match.
- **Implementation note:** `monte_carlo.py:107-122` builds the outputs; `monte_carlo.py:97-99`
  partitions runs into 200 blocks for the band.
- **Conditions of satisfaction:** no exception for `runs ∈ {1, 50, 199, 4000}`; `score_matrix`
  and `total_goals_dist` sum to 1±0.01. *(Currently violated for `runs<200` — CB-1.)*
- **Alternative paths:** `rho=0` (pure Poisson) and `rho≠0` (Dixon-Coles importance weights).
- **Use cases:** UC-01. **References:** `monte_carlo.py:20-122`.

### REQ-002 — Uncertainty band semantics (Tier 4, specific)
- **Requirement:** the `uncertainty` band must represent **parameter** uncertainty
  (CV = `MC_PARAM_UNCERTAINTY`) and be approximately invariant to `MC_RUNS`.
- **User story:** As the operator reading a p05/p95 band, I expect it to widen with genuine
  model uncertainty, not to shrink merely because I raised the simulation count, *so that* a
  performance knob does not silently change betting output.
- **Implementation note:** docstring `monte_carlo.py:3-6` states the parameter-resample intent;
  `monte_carlo.py:96-105` implements blockwise frequencies over independently-sampled λ runs.
- **Conditions of satisfaction:** band width at `runs=20000` ≈ band width at `runs=200000` for
  fixed λ (within MC tolerance). *(Currently violated — CB-2; band shrinks ~1/√chunk.)*
- **Use cases:** UC-01, UC-02. **References:** `monte_carlo.py:3-6,30-31,96-105`.

## Functional section: Ensemble & goal-structure parity

### REQ-003 — Probability path and λ path degrade together (Tier 3, specific)
- **Requirement:** when Polymarket `market` is absent but `books`/`kalshi` are present, the MC λ
  produced by `blend_lambdas` must reflect the same priced sources that `blend_probs` used.
- **User story:** As the operator for a match that exists only on bookmaker feeds, I expect the
  scoreline/totals distribution to be consistent with the headline probabilities, *so that*
  Over/Under EV is not computed from a model-only goal structure that the probabilities ignored.
- **Implementation note:** `blend_probs` admits books/kalshi (`ensemble.py:20-23,43`);
  `blend_lambdas` builds candidates only from `market`+`model` (`ensemble.py:74-89`).
- **Conditions of satisfaction:** with `market=None, books=<probs>, model=<m>`, the λ output must
  differ from the model-only λ in the direction of the book-implied λ. *(Currently violated — CB-3.)*
- **Use cases:** UC-03. **References:** `ensemble.py:13-102`.

### REQ-009 — Weight defaults match config (Tier 3, specific)
- **Requirement:** weight defaults referenced inline must equal `config.ENSEMBLE_WEIGHTS`.
- **Implementation note:** `ensemble.py:33` uses `0.2`; `config.py:45` defines whale `0.15`.
- **Conditions of satisfaction:** the literal fallback equals the config value. *(Violated — OF-8.)*
- **Use cases:** UC-02. **References:** `ensemble.py:33`, `config.py:45`.

## Functional section: Value detection & staking

### REQ-004 — Over/Under push semantics (Tier 3, specific)
- **Requirement:** Over/Under probabilities must satisfy `p_over + p_under + p_push = 1`, with
  `p_push = P(total = line)` for integer lines and 0 for `.5` lines.
- **Conditions of satisfaction:** `.5` line ⇒ `p_over+p_under=1`; integer line ⇒ `p_under` is
  `P(total<line)`, not `1-p_over`. *(Integer case currently violated — CB-4.)*
- **Use cases:** UC-02. **References:** `value_betting.py:132-153`.

### REQ-005 — Comparable uncertainty across markets (Tier 3, specific)
- **Requirement:** σ used for value detection must be derived comparably; a market lacking an MC
  band must not get a systematically smaller σ than a 1X2 row at equal edge.
- **Conditions of satisfaction:** at equal `(p_model, p_market, n_books)`, a totals row must not
  reach `is_value` at a strictly lower true confidence than a 1X2 row. *(At-risk — CB-5.)*
- **Implementation note:** totals pass `band_width=None` ⇒ default `bw=0.05` (`value_betting.py:38,148-150`).
- **Use cases:** UC-02. **References:** `value_betting.py:35-39,66-91`.

### REQ-006 — Commensurable whale exposure (Tier 3, specific)
- **Requirement:** whale exposure aggregation must combine a single unit (USD).
- **Conditions of satisfaction:** holder contributions and trade contributions are both in USD
  before summation. *(Currently violated — share-count + USD, CB-6.)*
- **Use cases:** UC-06. **References:** `whale_scoring.py:85-130`.

### REQ-010 — Data-quality gate suppresses value (Tier 3, specific)
- **Requirement:** a red tier must clear value flags and zero recommendations; amber scales
  stakes by 0.5; green keeps them.
- **Conditions of satisfaction:** `apply_quality_gate(..., tier=red)` ⇒ `value_bets == []` and
  every `is_value` False; amber halves `stake_pct`. *(Satisfied — covered by functional test.)*
- **Use cases:** UC-02. **References:** `value_betting.py:198-214`, `data_quality.py:62-66`.

## Functional section: Data quality & calibration

### REQ-007 — Team identity resolution avoids false positives (Tier 3, specific)
- **Requirement:** team matching must not bind two different teams via substring containment.
- **Conditions of satisfaction:** no two canonical names in the 48-team field match by
  `a in b`; result/odds alignment binds only the intended fixture. *(At-risk — CB-/OF-5.)*
- **Use cases:** UC-05. **References:** `calibration.py:30-33`, `odds_client.py:117-120`.

### REQ-008 — Market-agreement considers all outcomes (Tier 3, specific)
- **Requirement:** the data-quality market-agreement factor must reflect disagreement across all
  three outcomes, not only `team1_win`.
- **Conditions of satisfaction:** sources agreeing on home-win but disagreeing on draw/away do
  not score maximal agreement. *(Currently violated — OF-6.)*
- **Use cases:** UC-04. **References:** `data_quality.py:24-35`.

### REQ-011 — Polymarket Yes-price identification (Tier 3, specific)
- **Requirement:** `extract_1x2` must select the Yes price irrespective of outcome ordering.
- **Conditions of satisfaction:** a market listed `[No, Yes]` yields the same probability as
  `[Yes, No]`. *(Assumption-bound — parsing fidelity, OF/PD-3.)*
- **Use cases:** UC-01. **References:** `polymarket_client.py:98-119`.

### REQ-012 — Calibration leakage & timezone (Tier 3, specific)
- **Requirement:** calibration must use only pre-kickoff snapshots and be timezone-correct.
- **Conditions of satisfaction:** post-kickoff snapshots excluded; Z-suffix parsed as UTC.
  *(Satisfied — covered by functional + repo tests.)*
- **Use cases:** UC-05. **References:** `calibration.py:103-158`.

## Architectural guidance

### REQ-AG1 — Graceful degradation (architectural-guidance)
Every external source must degrade to a status-tagged fallback rather than raising; the pipeline
must complete with any subset of sources. Basis: pervasive `status` handling, pipeline warnings.

### REQ-AG2 — Conservative-by-default staking (architectural-guidance)
Staking must remain defensive: quarter-Kelly, per-tier caps, per-match correlation cap, portfolio
cap. Basis: `value_betting.py:25-32,217-235`.
